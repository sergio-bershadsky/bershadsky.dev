# Second Brain: The Memory Layer

Your second brain has skills and hooks. But ask it "how many decisions did we make last month?" and it has to read every file. There's a better way.

Skills process things. Hooks check things. But who *remembers* what's been done? That's the tracking layer—your second brain's memory.

## The Problem with Markdown Alone

Markdown is wonderful for reading. It's terrible for querying.

Imagine asking: "Show me all decisions from December that are still pending." With markdown files, Claude has to:

1. Open every decision file
2. Parse the content
3. Extract dates and statuses
4. Filter and sort

That's fifty file reads for a simple question. Now imagine asking the same question of a structured data file:

```
MARKDOWN FILES              STRUCTURED DATA
──────────────────         ──────────────────
Read 50 files               Read 1 file
Parse each one              Filter instantly
Hope for consistency        Know the format
5 seconds                   0.1 seconds
```

The difference isn't just speed—it's reliability. When your data has structure, Claude can query it confidently. When it's scattered across markdown, every query is an adventure.

## What Makes Good Structured Data?

Before we build anything, let's establish what we're aiming for:

- **Human-readable** — You'll edit this manually sometimes
- **Machine-parseable** — Claude can query it reliably
- **Version-controlled** — Git tracks every change
- **Simple** — No database servers, just files

The answer is YAML for data, JSON for schemas:

```
YAML                        JSON
────                        ────
Human-friendly format       Strict validation
Easy to edit manually       Perfect for schemas
Comments allowed            Claude knows it well

Use both together:
├── config.yaml         (your settings)
├── records.yaml        (your actual data)
└── schema.json         (validation rules)
```

YAML is forgiving and readable. JSON Schema validates the structure. Together they give you a database that lives in your git repository.

## The Four Principles of Schema Design

These principles will save you from overengineering—the graveyard where most tracking systems go to die.

### Principle 1: Start Minimal

Every field you add is a field you must maintain. Start with the absolute minimum:

```
BAD: Overengineered from day one
────────────────────────────────
decisions:
  - id: "decision-uuid-123-abc"
    created_at: "2024-01-15T14:30:00Z"
    updated_at: "2024-01-15T14:35:00Z"
    created_by: "user-uuid-456"
    status_history:
      - status: "draft"
        changed_at: "..."

GOOD: Just what you need
────────────────────────
decisions:
  - number: 1
    title: "Use Redis for caching"
    status: proposed
    created: 2024-01-15
```

The minimal version answers every question you'll actually ask. The overengineered version answers questions nobody's asking yet.

### Principle 2: Every Field Answers a Question

Before adding any field, ask: "What question does this answer?"

```
DECISION RECORD
───────────────
number: 1           → "Which decision?"
title: "..."        → "What was decided?"
status: proposed    → "Where in the process?"
created: 2024-01-15 → "When did this start?"
file: "path/to.md"  → "Where's the detail?"
```

If you can't articulate the question, you don't need the field.

### Principle 3: Denormalize Thoughtfully

This isn't SQL. Some duplication is not just okay—it's preferable:

```
NORMALIZED (requires lookups)
─────────────────────────────
discussion:
  id: 1
  participant_ids: [101, 102]

participants:
  - id: 101
    name: "Alice"
  - id: 102  
    name: "Bob"

DENORMALIZED (self-contained)
─────────────────────────────
discussion:
  participants:
    - Alice
    - Bob
```

The denormalized version is what you want. Each record stands alone. Claude doesn't need to chase references across files.

### Principle 4: Use Obvious Names

Your future self and Claude will thank you:

```
BAD                     GOOD
───                     ────
n: 1                    number: 1
t: "Thing"              title: "Thing"
s: "p"                  status: proposed
d: "2024-01-15"         created: 2024-01-15
```

Abbreviations save bytes and cost hours.

## Building Your Tracking System

Now let's build something real. Here's what most second brains need to track:

```
CORE ENTITIES
─────────────────────────────────────────
DECISIONS        DISCUSSIONS      REPORTS
─────────        ───────────      ───────
number           date             period
title            topic            type
status           participants     file
created          file             generated
file             source
author
```

Decisions capture choices. Discussions capture conversations. Reports capture what's been generated.

### Recommended Folder Structure

```
.claude/data/
├── config.yaml              # Static settings
├── decisions/
│   ├── schema.json          # Validation rules
│   └── records.yaml         # All decisions
├── discussions/
│   ├── schema.json
│   ├── 2024-01.yaml         # Monthly partitions
│   └── 2024-02.yaml
└── reports/
    └── tracking.yaml
```

Why partition discussions by month? Files stay manageable, archiving is natural, and date-range queries are obvious.

### Example: Decision Records

Here's what a real decision tracking file looks like:

```yaml
# decisions/records.yaml

last_number: 15

records:
  # Infrastructure decisions
  - number: 1
    title: Use Cloud Provider X
    status: implemented
    created: 2024-01-10
    file: docs/decisions/ADR-0001-cloud-provider.md
    author: Alice

    created: 2024-01-15
```

Simple. Queryable. Human-readable.

### Example: Discussion Records

```yaml

    - Alice
```

Each discussion is self-contained. No cross-references needed.

## Schema Validation

Validation sounds boring until you realize it prevents data corruption. Here's a simple JSON Schema for decisions:

```json
{
  "type": "object",
  "properties": {
    "number": { "type": "integer", "minimum": 1 },
    "title": { "type": "string", "minLength": 1 },
    "status": { 
      "type": "string",
      "enum": ["draft", "proposed", "accepted", "implemented", "rejected"]
    },
    "created": { "type": "string", "format": "date" },
    "file": { "type": "string", "pattern": "^docs/decisions/" }
  },
  "required": ["number", "title", "status", "created", "file"]
}
```

When Claude writes data, the validation flow is:

```
VALIDATION FLOW
───────────────────────────────────────────
Load data → Make changes → Validate → Save
                              ↓
                         If invalid:
                         Show error
                         Don't save
                         Fix and retry
```

This catches typos before they become data quality issues.

## Querying Your Data

Most queries are simple—just read the YAML:

- "What's the latest decision?" → Read records.yaml, find highest number
- "Who participated in this discussion?" → Read the file, it's right there
- "What's the status of ADR-5?" → Find number: 5, read status

For complex queries, you might write a small helper:

```python
import yaml

def get_decisions(status=None, after_date=None):
    with open('.claude/data/decisions/records.yaml') as f:
        data = yaml.safe_load(f)
    
    records = data.get('records', [])
    
    if status:
        records = [r for r in records if r['status'] == status]
    if after_date:
        records = [r for r in records if r['created'] > after_date]
    
    return records
```

But honestly? For most second brains, Claude just reads the YAML directly. The structure makes it obvious.

## Planning for Evolution

Your schema will change. That's not a bug—it's expected. Here's how to handle it:

**Approach 1: Optional new fields**

```yaml
# v1 → v2: Just add optional fields
# Old records work fine without them

v1: { number, title, status }
v2: { number, title, status, tags }  # tags is optional
```

**Approach 2: Version tracking**

```yaml
# In your config.yaml
meta:
  schema_version: "1.0.0"
  last_migration: 2024-01-15
```

**Approach 3: Migration scripts**

```python
# migrate-v1-to-v2.py
import yaml

with open('data/decisions/records.yaml') as f:
    data = yaml.safe_load(f)

for record in data['records']:
    if 'tags' not in record:
        record['tags'] = []

data['meta']['schema_version'] = '2.0.0'

with open('data/decisions/records.yaml', 'w') as f:
    yaml.dump(data, f)
```

The key is being intentional. Changes should be deliberate, not accidental.

## The Memory Advantage

With structured tracking in place, your second brain transforms:

```
BEFORE TRACKING           AFTER TRACKING
───────────────           ──────────────
"Find all decisions"      "decisions.yaml"
→ scan 50 files           → one file read

"What's pending?"         "status: proposed"

"Stats this month?"       "group by date"
→ impossible              → simple query
```

This is the memory layer that makes everything else work. Skills create content. Hooks check conditions. But tracking remembers what happened—and that memory is what transforms a collection of files into a true second brain.
