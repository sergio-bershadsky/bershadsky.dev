# Second Brain: Meeting Memory

Three weeks ago, we definitely discussed the API changes. What did we decide? Nobody remembers. The transcript is... somewhere.

Meetings produce decisions. Without memory, those decisions evaporate. This article shows you how to turn every meeting into searchable, permanent context.

## The Meeting Memory Problem

Here's the reality of meeting knowledge:

```
MEETING KNOWLEDGE REALITY
───────────────────────────────────────────────────────────────
Meeting happens ───▶ Transcript exists ───▶ ??? ───▶ "What did we decide?"
                           │
                           ▼
                     Nobody reads it
```

Transcripts exist. But they're not enough:

- Too long to re-read
- No structure or summary
- Can't search for decisions
- Context buried in conversation

What we actually want:

```
MEETING KNOWLEDGE IDEAL
───────────────────────────────────────────────────────────────
Meeting ───▶ Transcript ───▶ Structured Doc ───▶ Decisions ───▶ Searchable
                │                  │                  │
                ▼                  ▼                  ▼
          Auto-processed      With context      Linked to ADRs
```

The goal: every meeting becomes a queryable record. "What did we decide about the API in the December planning meeting?" gets an instant answer.

## The Transcription Pipeline

Here's how meeting memory works:

```
TRANSCRIPTION PIPELINE
────────────────────────────────────────────────────────────
┌────────────┐     ┌────────────┐     ┌────────────┐
│  MEETING   │────▶│ TRANSCRIPT │────▶│  SECOND    │
│            │     │  SERVICE   │     │   BRAIN    │
└────────────┘     └────────────┘     └────────────┘
      │                  │                  │
      ▼                  ▼                  ▼
   Recording         Raw text          Structured
   captured         generated           document
```

You have two options for triggering processing:

**Automatic Processing** works like this: meeting ends, transcript appears, skill triggers automatically, you get a notification. Great for high-value meetings where you always want documentation.

**Triggered Processing** gives you more control: run `/transcribe list` to see unprocessed meetings, then `/transcribe [id]` to process specific ones. Better for learning what works and refining your output.

The recommended approach:

1. **Start with triggered** — Learn what works, refine the output
2. **Move to automatic for high-value meetings** — Weekly syncs, planning sessions, decision-heavy discussions
3. **Keep casual meetings manual** — Not everything needs processing

## The Discussion Document

When your second brain processes a meeting, it creates a structured document. Here's the template:

```yaml
---
date: 2024-01-15
participants:
  - Alice
  - Bob
  - Charlie
topic: Q1 Planning Session
status: documented
source: transcript
meeting_id: abc123
duration: 45 minutes
---
```

The document body follows a consistent structure:

```
DISCUSSION DOCUMENT STRUCTURE
─────────────────────────────────────────────────
SECTION              PURPOSE
───────              ───────
FRONTMATTER          Basic metadata, searchable
CONTEXT              Why this meeting happened
DISCUSSION           What was talked about
DECISIONS            What was concluded
ACTION ITEMS         What needs to happen
OPEN QUESTIONS       What wasn't resolved
```

Each section serves a specific purpose. The frontmatter makes meetings findable. The context explains why the meeting happened. Discussion points capture the flow of conversation. Decisions are the payoff—the conclusions that matter. Action items ensure follow-through. Open questions track what still needs resolution.

## The Transcribe Skill

Here's the complete skill definition:

```markdown
# Transcribe Skill

Process meeting transcripts into structured discussion documents.

## Arguments

/transcribe list           # Show unprocessed meetings
/transcribe [meeting-id]   # Process specific meeting
/transcribe all            # Process all pending

## Procedure

### Step 1: Fetch Transcript
Connect to transcription service via MCP.
Get raw transcript for specified meeting.

### Step 2: Check Processing Status
Look up meeting_id in tracking data.
If already processed, show existing document location.
If not, continue processing.

### Step 3: Extract Information

From the transcript, identify:
- Participants: Who was in the meeting
- Topics: Main subjects discussed
- Decisions: Conclusions reached
- Action Items: Tasks assigned with owners
- Open Questions: Unresolved items

### Step 4: Generate Document
Create markdown file following the discussion template.
Include all frontmatter metadata.

### Step 5: Identify Potential ADRs
Scan decisions for architectural or technical choices.
Present candidates to user for confirmation.

### Step 6: Update Tracking
Add meeting to processed meetings.
Link any created ADRs.
```

Here's what a session looks like:

```
You:    /transcribe list

Claude: ## Unprocessed Meetings (Last 7 Days)

        | Date       | Title              | Duration |
        |------------|--------------------|----------|
        | 2024-01-15 | Q1 Planning        | 45 min   |
        | 2024-01-14 | Tech Sync          | 30 min   |

You:    /transcribe q1-planning

Claude: Processing "Q1 Planning Session"...

        Created: docs/discussions/2024-01-15-q1-planning.md

        Found 3 decisions:
        1. "API v2 by mid-March" — Create ADR? [Y/N]
        2. "Postpone mobile app" — Create ADR? [Y/N]
        3. "Start hiring process" — Create ADR? [Y/N]
```

With meetings accumulating, your second brain becomes a queryable archive.

This is the power of meeting memory: tracing any decision back through the discussions that shaped it.

## Making It Stick

Consistency makes meeting memory valuable. Here are the rituals that work:

**Weekly rituals:**
- Process any unprocessed meetings
- Review action items from meetings
- Check for stale open questions

**After important meetings:**
- Immediately run `/transcribe`
- Create any obvious ADRs
- Share document link with participants

**Quality checks for each document:**
- All participants identified correctly?
- Key decisions captured?
- Action items have owners?
- Open questions make sense?
- Linked to relevant ADRs?

## The Compound Effect

The value of meeting memory compounds over time:

- **After 3 months:** A searchable archive of every significant discussion
- **After 6 months:** You can trace any decision back to the meeting that made it
- **After a year:** New team members can understand the entire project history

Every meeting you process adds to this foundation. Skip a few and you create gaps. Stay consistent and you build institutional memory that outlasts any individual team member.
