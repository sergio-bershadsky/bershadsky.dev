# Second Brain: Decision Intelligence

"Why do we use this framework?" The question hangs in the air. You were there when the decision was made. But the reasoning? Gone.

Decisions without context become arbitrary rules. Decision intelligence makes the "why" permanent and queryable.

## The Decision Decay Problem

Here's what happens to decisions over time:

```
DECISION DECAY TIMELINE
─────────────────────────────────────────────────────────
WEEK 1:   "We decided to use Redis because it handles
           our read patterns better than alternatives,
           and the team has experience with it."

MONTH 3:  "We use Redis. It was decided at some point."

YEAR 1:   "Why Redis? I don't know. That's just what we use."
```

The cost of decision decay:

- New team members ask "why?" constantly
- Old decisions get relitigated every quarter
- Tribal knowledge leaves when people do
- "We should switch to X" becomes an annual tradition

The symptoms are easy to recognize:

- Repeating the same debate every quarter
- "We tried that before" with no details
- Fear of changing anything
- Newcomers feeling excluded from history

What we want: "Why Redis?" returns an instant answer with full context, alternatives considered, and the discussion that led there.

## Anatomy of a Good Decision Record

A complete decision record captures everything future-you needs:

```
DECISION RECORD ANATOMY
─────────────────────────────────────────────────────────
┌───────────────────────────────────────┐
│  DECISION: Use Redis for Caching      │
├───────────────────────────────────────┤
│  NUMBER:   ADR-0015                   │
│  STATUS:   Implemented                │
│  DATE:     2024-01-15                 │
│  AUTHOR:   Alice                      │
├───────────────────────────────────────┤
│  CONTEXT                              │
│  API response times were too slow     │
│  for expected Q2 volume.              │
├───────────────────────────────────────┤
│  OPTIONS CONSIDERED                   │
│                                       │
│  1. Redis (CHOSEN)                    │
│     + Team experience                 │
│     + Handles read patterns           │
│                                       │
│  2. Memcached (rejected)              │
│     - No persistence option           │
│                                       │
│  3. In-memory (rejected)              │
│     - Multiple instances problem      │
├───────────────────────────────────────┤
│  CONSEQUENCES                         │
│  - Need Redis in production           │
│  - Team learns Redis patterns         │
├───────────────────────────────────────┤
│  RELATED                              │
│  - Discussion: 2024-01-10 Tech Sync   │
│  - Supersedes: ADR-0008               │
└───────────────────────────────────────┘
```

Each section serves a purpose:

```
ADR SECTION PURPOSE
─────────────────────────────────────────────────────────
SECTION          PURPOSE
───────          ───────
CONTEXT          Why did we need to decide anything?
OPTIONS          What did we actually consider?
CHOSEN           What did we pick and why?
REJECTED         Why NOT the alternatives?
CONSEQUENCES     What does this mean going forward?
RELATED          Where's the deeper context?
```

The rejected options are gold. When someone suggests Memcached next year, you don't debate. You show them: "We considered it. Here's why we didn't choose it. If circumstances changed, here's what would need to be true."

## From Conversation to Decision

Recognizing when a decision happens:

```
DECISION SIGNAL PATTERNS
─────────────────────────────────────────────────────────
PATTERN                      MEANING
───────                      ───────
"Let's go with..."           Decision made
"We're choosing..."          Decision made
"I think we should..."       Proposal (not yet decision)
"What about..."              Exploration (definitely not)
"Agreed, we'll use..."       Decision confirmed
```

The decision flow looks like this:

```
DECISION FLOW
─────────────────────────────────────────────────────────
EXPLORATION ──▶ PROPOSAL ──▶ DISCUSSION ──▶ DECISION
     │              │             │             │
     ▼              ▼             ▼             ▼
 "What options   "I propose   "Concerns?    "Agreed.
  do we have?"    Redis"      Trade-offs?"   Redis it is"
```

Not every decision needs an ADR. Use the reversibility test: if reversing this decision would be expensive (time, money, effort), it deserves an ADR. If you can change it in 10 minutes, probably not.

**ADR-worthy decisions:**
- Technology choices
- Architecture patterns
- Process changes
- Integration decisions
- Security approaches

**Just note these informally:**
- Variable naming
- Meeting schedules
- Temporary workarounds
- Formatting preferences

## Decision Categories

Organize decisions with numbered categories:

```
DECISION NUMBERING SCHEME
─────────────────────────────────────────────────────────
RANGE           CATEGORY
─────           ────────
0001-0999       Architecture & Infrastructure
1000-1999       Fork & Upstream
2000-2999       Feature & Product
3000-3999       Process & Workflow
```

Examples:

- ADR-0001: Cloud Platform Selection
- ADR-0015: Redis for Caching
- ADR-2001: Payment Integration Approach
- ADR-3001: Release Cadence

Categories make querying powerful: "Show me all infrastructure decisions" returns ADR-0001 through ADR-0999 instantly.

## Building the Decision Skill

Here's the skill definition:

```markdown
# ADR Skill

Create Architecture Decision Records from context.

## Arguments

/adr <category> <title>
- category: infra, feature, process
- title: Short description

## Procedure

### Step 1: Gather Context
Ask for or find:
- What problem are we solving?
- What options did we consider?
- What did we choose and why?

### Step 2: Identify Rejected Options
For each alternative not chosen:
- Why wasn't it selected?
- Under what conditions might we reconsider?

### Step 3: Document Consequences
- What becomes possible now?
- What becomes harder?

### Step 4: Link Related Items
- Connect to discussions that led here
- Reference superseded decisions

### Step 5: Assign Number
Based on category, get next available number.

### Step 6: Generate Document
Create markdown file with full template.
Location: docs/decisions/ADR-XXXX-slug.md
```

Decision states follow a lifecycle:

```
DECISION STATES
─────────────────────────────────────────────────────────
DRAFT ──▶ PROPOSED ──▶ ACCEPTED ──▶ IMPLEMENTED
              │
              ▼
           REJECTED (with reason)

ACCEPTED ──▶ SUPERSEDED (by newer ADR)
```

Here's what a session looks like:

```
You:    /adr infra redis-caching

Claude: Creating ADR for infrastructure decision: Redis Caching

        ## Context (required)
        What problem are we solving?

You:    API responses are too slow, need caching

Claude: ## Options Considered
        What alternatives did you evaluate?

You:    Redis, Memcached, and in-memory caching

Claude: [Asks about pros/cons for each]
        [Generates complete ADR]

        Created: docs/decisions/ADR-0015-redis-caching.md

        Ready to save? [Y/N]
```

## Decision Intelligence Queries

Once you have decisions documented, querying becomes powerful.

**Basic queries:**

```
You:    "Why do we use Redis?"

Claude: Based on ADR-0015 (Implemented, Jan 2024):

        Context: API response times were too slow for Q2 volume.

        We chose Redis because:
        - Team has production experience
        - Handles our read-heavy patterns well
        - Persistence options for session data

        Alternatives considered:
        - Memcached (rejected: no persistence)
        - In-memory cache (rejected: doesn't work across instances)
```

**Advanced queries:**

```
You:    "Show me all decisions from Q4"

Claude: Found 8 decisions from Q4 2024:
        - ADR-0015: Redis for Caching (Implemented)
        - ADR-0016: WebSocket Approach (Proposed)
        - ADR-2003: Notification System (Accepted)
        ...

You:    "What decisions did Alice author?"

Claude: Alice authored 5 decisions:
        - ADR-0015: Redis for Caching
        - ADR-0018: API Versioning Strategy
        ...
```

**Cross-referencing:**

```
You:    "What discussions led to ADR-0015?"

Claude: ADR-0015 references:
        - Discussion: 2024-01-10 Tech Sync
          [Topic: Caching strategies for API performance]

        - Discussion: 2024-01-08 Architecture Review
          [Topic: Infrastructure scaling options]
```

The compound value emerges when onboarding new team members:

```
New Member:  "Why is everything so complicated?"

You:         "Let me show you the decision history."

Claude:      [Returns timeline of decisions with context]

New Member:  "Oh, that actually makes sense now."
```

## Sample ADR

Here's a complete example:

```markdown
## ADR-0015: Redis for API Caching

### Status
Implemented

### Context
API response times averaging 800ms under load.
Q2 growth projections require sub-200ms responses.
Need a caching layer for frequently-accessed data.

### Decision
Use Redis as our caching layer.

### Options Considered

**Redis (Chosen)**
- Team has production experience
- Handles our read-heavy patterns well
- Persistence options for session data
- Good monitoring ecosystem

**Memcached**
- Simpler than Redis
- But: No persistence, less feature-rich
- Rejected: We might need pub/sub later

**In-Memory Cache**
- No additional infrastructure
- But: Doesn't work across multiple instances
- Rejected: We're scaling horizontally

### Consequences
- Need Redis in production infrastructure
- Team will learn Redis patterns
- Can leverage pub/sub for real-time features
- Another service to monitor and maintain

### Related
- Discussion: 2024-01-10-tech-sync-caching
- Supersedes: ADR-0008-local-cache-approach
```
