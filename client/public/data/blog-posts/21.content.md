# Team Communication Hub

Monday morning. 47 unread messages. Somewhere in there is something important. But where?

Team chat is a firehose. This article shows you how to build a communication hub that filters signal from noise and makes important things findable.

## The Communication Problem

Every team generates a constant stream of messages. Most are noise. But buried in that noise are decisions, commitments, and context that matter.

```
THE FIREHOSE
────────────
9:00  "Good morning!"
9:15  "Anyone seen the API docs?"
9:30  "Meeting moved to 2pm"
9:45  "We should use Redis for caching"  ← DECISION
10:00 "lol"
10:15 "Can someone review my PR?"
10:30 "Decided to postpone mobile app"   ← DECISION
11:00 "Coffee anyone?"
```

Sound familiar? The symptoms are universal:

- Important decisions buried in casual chat
- Context lost between conversations
- "I thought we agreed on..." syndrome
- New team members drowning in history

What we actually want is simple:

```
THE SUMMARY
───────────
Decisions: 2
- Use Redis for caching
- Postpone mobile app

Action items: 1
- PR review needed

Everything else: Filtered
```

The goal: transform chaotic streams into structured, searchable memory.

## Building the Hub

The communication hub brings together everything we've built so far. Here's the architecture:

```
TEAM COMMUNICATION HUB
─────────────────────────────────────────────────────
SOURCES              PROCESSING           OUTPUT
───────              ──────────           ──────

Team Chat ────┐                       
              │      ┌──────────┐     
Direct Msgs ──┼─────▶│  SECOND  │────▶ Summaries
              │      │  BRAIN   │     
Channels ─────┘      │  SKILLS  │────▶ Decisions
                     └──────────┘     
                          │           
                          └──────────▶ Actions
```

Each component comes from what you've already learned:

**MCP Connection** (from Part 4) gives Claude access to your team chat. Whether it's Slack, Discord, or something else, the connection handles authentication and data access.

**Recap Skill** (from Part 5) processes messages on demand. You trigger it when you want a summary, and it knows exactly how to categorize what it finds.

**Tracking Layer** (from Part 8) remembers what's been processed. No duplicate summaries, no missed messages.

### Channel Configuration

Not all channels are equal. Configure which ones matter:

```yaml
# config.yaml
communication:
  channels:
    - id: "channel-123"
      name: "#engineering"
      type: public
      priority: high

    - id: "channel-456"
      name: "#general"
      type: public
      priority: low

    - id: "dm-789"
      name: "DM with Alice"
      type: direct
      priority: high
```

High-priority channels get full processing. Low-priority ones get lighter treatment—maybe just decision extraction, no detailed summaries.

## The Recap Skill

This is the core of your communication hub. Here's the complete skill definition:

```markdown
# Slack Recap Skill

Generate a summary of recent team activity.

## Arguments

/recap [hours]
- hours: Number of hours to look back (default: 24)

## Procedure

### Step 1: Calculate Time Range
Determine the cutoff timestamp based on hours argument.

### Step 2: Fetch Channel History
For each configured channel, get messages since cutoff.
Use the MCP connection to team chat.

### Step 3: Filter by Timestamp
Only include messages within the time range.

### Step 4: Categorize Messages

Look for:
- Decisions: "decided", "agreed", "will do", "going with"
- Blockers: "blocked", "waiting", "can't proceed"
- Action items: "will", "need to", "should", "TODO"
- Questions: Contains "?", "thoughts", "what do you think"

### Step 5: Generate Summary

Format the output with sections for decisions, blockers,
action items, and key discussions.

### Step 6: Update Tracking
Mark this time range as processed.
```

Triggering is simple:

```
/recap              # Last 24 hours
/recap 8            # Last 8 hours (since this morning)
/recap 72           # Last 3 days (after vacation)
```

The skill handles the complexity. You just ask for what you need.

## Decision Extraction

The most valuable part of any communication stream is the decisions. Here's how to recognize them:

```
SIGNAL WORDS                 EXAMPLES
────────────                 ────────
"decided to"                 "We decided to use Redis"
"agreed on"                  "Team agreed on the timeline"
"going with"                 "Going with option B"
"will use"                   "Will use the new API"
"settled on"                 "Settled on weekly releases"
```

When the skill finds a decision, it extracts the key information:

```
FROM MESSAGE TO RECORD
──────────────────────────────────────────────────────
MESSAGE:
"After discussing the options, we decided to use
 Redis for the caching layer. @alice will set it up."

EXTRACTED:
─────────
Decision: Use Redis for caching layer
Owner: Alice
Context: Options discussion
Date: 2024-01-15
Source: #engineering

NEXT STEP:
"This looks like a decision. Create an ADR? [Y/N]"
```

When you confirm, it creates a proper decision record:

```yaml
- number: 15
  title: Use Redis for caching layer
  status: proposed
  created: 2024-01-15
  source: chat-message
  source_channel: "#engineering"
  file: docs/decisions/ADR-0015-redis-caching.md
```

Chat decisions become real documentation automatically.

## Privacy Considerations

Not everything should be captured. Here's the principle: capture work, not workers.

```
CAPTURE VS SKIP
───────────────────────────────────────────
CAPTURE                 SKIP
───────                 ────
Technical decisions     Personal conversations
Project discussions     HR topics
Action items            Salary discussions
Blockers                Private DMs (usually)
```

Configure exclusions explicitly:

```yaml
communication:
  skip_channels:
    - "#hr-confidential"
    - "#leadership-private"

  skip_patterns:
    - "salary"
    - "compensation"
    - "confidential"

  anonymize_dms: true
```

Before deploying, get team buy-in:

> "Hey team, I'm setting up automated recaps for our channels. It summarizes decisions and action items—nothing personal. Let me know if you want any channels excluded."

Transparency builds trust. When people know what's being captured and why, they're usually supportive. Nobody wants to lose important decisions to the chat scroll.

## The Communication Advantage

With your communication hub in place, you gain:

- **Never miss a decision** — They're extracted and tracked automatically
- **Instant context** — Ask about any topic and get the relevant history
- **Prepared meetings** — Know what's been discussed before you walk in
- **Onboarding help** — New team members can catch up in minutes, not days

Your second brain becomes the team member who actually reads all the messages. The one who remembers what was decided, who said what, and where to find the details.
