# Sharing Knowledge: Plugins

> You've built skills and hooks that work perfectly for you. Wouldn't it be nice if others could use them too?

---

## The Opening

> "Your teammate asks how you automated your weekly reports. You could explain for 30 minutes, or you could say: 'Just install my plugin.'"

Plugins turn your personal automation into team infrastructure. One package, instant capability.

---

## From Personal to Shareable

Here's how the journey typically unfolds:

```
STAGE 1: Personal automation
         "I have a /standup skill that works great"

STAGE 2: Team interest
         "Can I get that standup thing you have?"

STAGE 3: Sharing challenge
         "Here's 5 files, put them here, rename this..."

STAGE 4: Plugin solution
         "Install my-standup-plugin. Done."
```

**What a plugin is:**

> A plugin is a packaged collection of skills, hooks, and configuration that can be installed and used immediately.

**Why plugins matter:**
- No copy-paste instructions
- Version controlled
- Easy updates
- Consistent experience

**The mental shift:**

> You're not just automating your work anymore. You're building tools that help your entire team.

And honestly? Publishing your first plugin feels like releasing music on Spotify. Except the audience is other developers, the music is productivity automation, and nobody will judge your album art.

## Plugin Structure

Every plugin follows a standard anatomy:

```
my-second-brain-plugin/
├── plugin.json           # Metadata & configuration
├── skills/               # Shareable skills
│   ├── standup/
│   │   └── SKILL.md
│   └── recap/
│       └── SKILL.md
├── hooks/                # Shareable hooks
│   ├── freshness.md
│   └── sidebar-check.py
├── agents/               # Optional: specialized agents
│   └── reviewer/
│       └── AGENT.md
└── README.md             # Documentation
```

**The plugin manifest (plugin.json)** is your plugin's identity card:

```json
{
  "name": "team-productivity",
  "version": "1.0.0",
  "description": "Common skills and hooks for team workflows",

  "skills": [
    "skills/standup",
    "skills/recap"
  ],

  "hooks": [
    "hooks/freshness.md",
    "hooks/sidebar-check.py"
  ],

  "configuration": {
    "team_chat_channel": {
      "description": "Default channel for team updates",
      "default": "#general"
    }
  }
}
```

**Key components at a glance:**

```
COMPONENT          PURPOSE
─────────          ───────
plugin.json        Tells Claude what's included
skills/            Reusable procedures
hooks/             Automatic behaviors
agents/            Specialized assistants
README.md          How to use this plugin
```

## Creating Your First Plugin

### Step 1: Organize existing work

Start with skills and hooks you've already built:

```
Before:                          After:
your-second-brain/               team-productivity-plugin/
├── skills/                      ├── plugin.json
│   └── standup/                 ├── skills/
├── hooks/                       │   └── standup/
│   └── freshness.py             ├── hooks/
                                 │   └── freshness.py
                                 └── README.md
```

### Step 2: Create the manifest

```json
{
  "name": "team-productivity",
  "version": "1.0.0",
  "description": "Daily standup and freshness checking",

  "skills": ["skills/standup"],
  "hooks": ["hooks/freshness.py"],

  "author": "Your Name",
  "repository": "github.com/you/team-productivity-plugin"
}
```

### Step 3: Add configuration options

Make the plugin adaptable to different team needs:

```json
{
  "configuration": {
    "standup_format": {
      "description": "Output format for standups",
      "options": ["brief", "detailed"],
      "default": "brief"
    },
    "freshness_days": {
      "description": "Days before content is considered stale",
      "type": "number",
      "default": 30
    }
  }
}
```

### Step 4: Write documentation

Your README is the user's first impression:

```markdown
# Team Productivity Plugin

## Installation
claude plugin install github.com/you/team-productivity-plugin

## Skills

### /standup
Generates your daily standup update.

### /recap
Summarizes activity since last session.

## Configuration
- `standup_format`: "brief" or "detailed"
- `freshness_days`: Number of days before stale warning
```

### Step 5: Test locally

Before sharing with anyone:

```bash
# Install your own plugin locally
claude plugin install ./team-productivity-plugin

# Test each skill
/standup
/recap

# Verify hooks fire correctly
```

## Distribution Options

You have three main ways to share your plugins:

```
Option 1: Git repository (simplest)

Pros:
✓ Version control built-in
✓ Easy to update
✓ No registry needed

Cons:
✗ Requires repo access
✗ Manual versioning

Install:
claude plugin install github.com/org/plugin-name
```

```
Option 2: Direct sharing

Pros:
✓ Works offline
✓ No external dependencies

Cons:
✗ No automatic updates
✗ Version confusion

Install:
claude plugin install /path/to/plugin-folder
```

```
Option 3: Team registry (advanced)

Pros:
✓ Central discovery
✓ Access control
✓ Audit trail

Cons:
✗ Setup overhead
✗ Maintenance burden

Install:
claude plugin install @team/plugin-name
```

**Recommendation for most teams:** Start with a private Git repository. It's easy to share within your team, version control is automatic, and you can pull updates with one command.

## Plugin Best Practices

### Make configuration flexible

```json
// BAD: Hardcoded values
"channel": "#engineering"

// GOOD: Configurable with sensible default
"channel": {
  "description": "Channel for updates",
  "default": "#general"
}
```

### Document thoroughly

Every skill should have clear usage examples:

```markdown
## /standup

Generates your daily standup update.

### Usage
/standup              # Brief format
/standup detailed     # Full format

### Configuration
- `format`: "brief" or "detailed"

### Example Output
**Yesterday:** Fixed auth bug, reviewed 2 PRs
**Today:** Working on caching layer
**Blockers:** Waiting on API specs
```

### Version meaningfully

Follow semantic versioning:

```
1.0.0 → Initial release
1.1.0 → Added new skill
1.2.0 → New configuration option
2.0.0 → Breaking change (different output format)
```

### Handle missing dependencies

Be graceful when requirements aren't met:

```markdown
## Requirements

This skill requires:
- MCP connection to team chat
- Access to task tracker

If these aren't configured, the skill will:
- Warn about missing connections
- Provide partial output where possible
```

## Team Plugin Strategies

Different situations call for different approaches:

```
PATTERN                USE CASE
───────                ────────
Team conventions       Standard skills for all projects
Project-specific       Custom workflows for one project
Personal toolkit       Your own productivity hacks
```

**Composing plugins** lets you layer functionality:

```bash
# Install team baseline
claude plugin install github.com/team/conventions

# Add project-specific
claude plugin install github.com/team/project-alpha

# Add personal preferences
claude plugin install github.com/you/personal-toolkit
```

**Plugin hierarchy** determines which takes precedence:

```
PROJECT PLUGINS (highest priority)
        │
        ▼
   TEAM PLUGINS
        │
        ▼
 PERSONAL PLUGINS (lowest priority)
```

When skills have the same name, the higher-priority plugin wins. This lets projects override team defaults when needed.
